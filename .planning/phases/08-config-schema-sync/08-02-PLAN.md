<objective>
Add a CI gate that proves `appsettings.Production.template.json` continues to validate against `appsettings.schema.json` on every push and PR. Catches drift the moment a developer edits one file without updating the other.

Purpose: Schema becomes useless if template can drift undetected. CI is the single enforcement point for D-02 (schema = source of truth).
Output: Workflow step that runs in seconds, fails loudly with line+path on mismatch.

  <behavior>
    - On any push/PR, the new step runs and exits 0 when template validates.
    - On a deliberate breaking edit (e.g., add an unknown required key to schema OR remove a required key from template), the step exits non-zero and the build fails.
    - The CI log surfaces `Test-Json`'s native error message including JSON path and line number.
  </behavior>
  <action>
    1. Open `.github/workflows/ci.yml`. Confirm the build job runs on `runs-on: windows-latest` (required — Linux runners lack `Test-Json -SchemaFile`).
    2. Locate the step `- name: Restore .NET dependencies` (or equivalent first .NET step). Insert a NEW step IMMEDIATELY AFTER restore and BEFORE `Build solution`:
       ```yaml
             - name: Validate appsettings.Production.template.json against schema
               shell: pwsh
               run: |
                 $ErrorActionPreference = 'Stop'
                 $errors = @()
                 try {
                   $valid = Test-Json `
                     -Path 'src/PassReset.Web/appsettings.Production.template.json' `
                     -SchemaFile 'src/PassReset.Web/appsettings.schema.json' `
                     -ErrorVariable errors `
                     -ErrorAction SilentlyContinue
                 } catch {
                   $valid = $false
                   $errors = @($_.Exception.Message)
                 }
                 if (-not $valid) {
                   Write-Host '::error::Schema validation failed for appsettings.Production.template.json'
                   foreach ($e in $errors) { Write-Host "::error::$e" }
                   exit 1
                 }
                 Write-Host 'Schema validation passed.'
       ```
    3. **Do NOT replace** any existing step — only insert. Preserve the existing `dotnet build` / `dotnet test` chain.
    4. If the workflow uses a reusable workflow (`uses:`) for build, add the step in the calling workflow before the reusable invocation, OR pass through to the reusable workflow if that's the pattern. Check `.github/workflows/tests.yml` for the reusable pattern (referenced in CLAUDE.md).
    5. Verify locally by running the same command in pwsh:
       ```pwsh
       Test-Json -Path src/PassReset.Web/appsettings.Production.template.json -SchemaFile src/PassReset.Web/appsettings.schema.json
       ```
       Must print `True`. Then deliberately break the schema (add a required key not in template) and verify the command prints `False` with a clear path/reason — revert the break before committing.
    6. Commit with message: `ci(deploy): validate appsettings template against schema in CI`.
  </action>
  <verify>

  </verify>
