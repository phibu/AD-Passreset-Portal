<objective>
Add a pre-check in `PasswordChangeProvider.PerformPasswordChangeAsync` that compares
the user's `pwdLastSet` against the domain `minPwdAge`. If the time since the last change
is less than the domain minimum, return `ApiErrorItem(ApiErrorCode.PasswordTooRecentlyChanged, ...)`
with a minute-level remaining-time message BEFORE invoking the AD password-change call.

This eliminates the current `UnauthorizedAccessException`/`E_ACCESSDENIED` that surfaces
as a generic UI crash when users perform two consecutive password changes within the
domain's minimum password age window. Per CONTEXT D-05 the existing `COMException` catch
block at lines 470-518 is NOT removed — it remains a defense-in-depth floor for races,
replication lag, and domain-policy edge cases.

Purpose: Closes ROADMAP.md Phase 7 Success Criterion #4 (gh#36).

Output: New private `PreCheckMinPwdAge` helper, call-site in `PerformPasswordChangeAsync`,
xUnit tests, removal of the stale "min-pwd-age trips a generic error" entry from
`docs/Known-Limitations.md`.
</objective>

    1. Add private method below the existing `AcquireDomainMinPasswordAge` helper (near line 647):

       ```csharp
       /// <summary>
       /// STAB-004 pre-check. Returns non-null when the user's pwdLastSet is newer than the
       /// domain minPwdAge. Matches the existing service-account-with-fallback pattern
       /// (D-06) by calling AcquirePrincipalContext(). Never throws — returns null on any
       /// unreadable state so the existing COMException catch (lines 470-518) remains the
       /// defense-in-depth floor (D-05).
       /// </summary>
       private ApiErrorItem? PreCheckMinPwdAge(string username)
       {
           try
           {
               var minAge = AcquireDomainMinPasswordAge();
               if (minAge == null || minAge.Value <= TimeSpan.Zero) return null;

               using var ctx = AcquirePrincipalContext();
               using var user = UserPrincipal.FindByIdentity(ctx, username);
               if (user == null) return null;

               var lastSet = user.LastPasswordSet;
               if (lastSet == null) return null; // must-change-at-next-logon — exempt

               var elapsed = DateTime.UtcNow - lastSet.Value.ToUniversalTime();
               if (elapsed >= minAge.Value) return null;

               var remaining = minAge.Value - elapsed;
               int remainingMinutes = Math.Max(1, (int)Math.Ceiling(remaining.TotalMinutes));
               int policyMinutes = Math.Max(1, (int)Math.Ceiling(minAge.Value.TotalMinutes));
               int elapsedMinutes = Math.Max(0, (int)Math.Floor(elapsed.TotalMinutes));

               var message =
                   $"Password was changed {elapsedMinutes} minute(s) ago; " +
                   $"AD policy requires {policyMinutes} minute(s) between changes " +
                   $"({remainingMinutes} minute(s) remaining).";

               _logger.LogWarning(
                   "STAB-004 pre-check blocked consecutive change for {User}: elapsed={Elapsed} minAge={MinAge}",
                   username, elapsed, minAge.Value);

               return new ApiErrorItem(ApiErrorCode.PasswordTooRecentlyChanged, message);
           }
           catch (Exception ex)
           {
               _logger.LogWarning(ex,
                   "STAB-004 pre-check failed for {User}; falling through to bind and catch path",
                   username);
               return null;
           }
       }
       ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Authenticated user → PasswordChangeProvider | User-typed creds bind to AD; pre-check now reads pwdLastSet via service-account-with-fallback |
| PreCheckMinPwdAge → AD LDAP | New LDAP read path (pwdLastSet) on the hot password-change path |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-03 | Information Disclosure | PreCheckMinPwdAge error message | mitigate | Message reveals that a prior change occurred (implicit via remaining-time value). This is acceptable — the user making the pre-check attempt is already authenticated (pre-check runs AFTER auth succeeds per insertion-point in Task 1 action step 2). No cross-account enumeration: pre-check only runs for the `username` the user typed AND for which authentication already succeeded. |
| T-07-04 | Denial of Service | New LDAP read on hot path | accept | One additional LDAP query per password change attempt. `AcquirePrincipalContext()` is the same context used by `GetUsersInGroup`/`GetUserEmail` (already on the request path). Pre-check fails-open on any exception (returns null) so a flaky DC does not block legitimate changes beyond the existing rate limiter. |
| T-07-05 | Elevation of Privilege | Service-account credential use in pre-check | mitigate | Pre-check uses `AcquirePrincipalContext()` which already encapsulates service-account-vs-bound-user selection (D-06). No new credential path; no new permission required. Service account needs the same pwdLastSet read it already has for expiry-notification scanning. |
| T-07-06 | Tampering | Removing the existing COMException catch would regress BUG-004 | mitigate | Acceptance criterion on Task 1 asserts `E_ACCESSDENIED` mapping remains in the catch block. Task 2 Test 6 fails if the catch block is removed — permanent regression guard. |
</threat_model>
