<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| server → client (`/api/password/policy`) | Anonymous GET endpoint — data flows to pre-auth user context; anything returned is public-visible |
| user → form (pre-submit) | User has not yet authenticated; information shown here is available to anyone who loads the page |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-04-01 | Information Disclosure | Panel exposes domain-level password policy pre-auth | accept | Documented in `docs/Known-Limitations.md`. Domain default policy is low-sensitivity; organizations concerned can disable via `ShowAdPasswordPolicy=false`. D-17 defers FGPP (per-user PSO), which is the variant that would leak user-specific policy. |
| T-10-04-02 | Spoofing | Stale/cached policy misleads user | mitigate | `usePolicy` hook re-fetches on mount; policy is short-lived in React state. No long-term client cache. Test 4 in Task 1 ensures the loading state is handled (skeleton render) so users never see a stale pre-loaded value. |
| T-10-04-03 | Tampering | Accidental re-introduction of a disclosure widget (`<Collapse>`, `<Accordion>`) hides the panel | mitigate | Task 2 Test 4 asserts no `aria-expanded` / `aria-controls` attributes appear on the panel — regression gate. |
| T-10-04-04 | Denial of Service | `/api/password/policy` request on every page load | accept | Endpoint is anonymous and cheap. Already current production behavior when operators enabled the panel; only the default flipped. |
| T-10-04-05 | Elevation of Privilege | Server change to anonymous policy endpoint | — | No server change in this plan (D-18 confirmed). Threat not applicable to Phase 10. |
</threat_model>
