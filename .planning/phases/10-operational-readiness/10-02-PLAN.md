    2. IMMEDIATELY AFTER the existing STAB-001 URL announcement block (the `Write-Ok "PassReset reachable at ..."` lines around 921/937), INSERT the following post-deploy verification block. It must run after IIS AppPool recycle and before the script exits with success:
       ```powershell
       # ----- STAB-019: post-deploy verification -----
       if (-not $SkipHealthCheck) {
           $baseUrl = if ($CertThumbprint -and $HttpsPort) {
               "https://${hostHeader}:${HttpsPort}"
           } else {
               "http://${hostHeader}:${selectedHttpPort}"
           }

           $maxAttempts = 10
           $attempt     = 0
           $ok          = $false
           $lastHealth  = $null
           $lastSettings = $null

           Write-Step "Verifying deployment at $baseUrl (up to $maxAttempts x 2s)"

           do {
               Start-Sleep -Seconds 2
               $attempt++
               try {
                   $lastHealth   = Invoke-WebRequest -Uri "$baseUrl/api/health"   -UseBasicParsing -TimeoutSec 5 -ErrorAction Stop
                   $lastSettings = Invoke-WebRequest -Uri "$baseUrl/api/password" -UseBasicParsing -TimeoutSec 5 -ErrorAction Stop
                   if ($lastHealth.StatusCode -eq 200 -and $lastSettings.StatusCode -eq 200) {
                       $ok = $true
                   }
               } catch {
                   Write-Warning ("Attempt {0}/{1}: {2}" -f $attempt, $maxAttempts, $_.Exception.Message)
               }
           } while (-not $ok -and $attempt -lt $maxAttempts)

           if (-not $ok) {
               $bodySnippet = if ($lastHealth) { $lastHealth.Content } else { "(no response)" }
               Write-Error ("Post-deploy health check failed after {0} attempts. Last /api/health response: {1}" -f $maxAttempts, $bodySnippet)
               exit 1
           }

           try {
               $body  = $lastHealth.Content | ConvertFrom-Json
               $ad    = $body.checks.ad.status
               $smtp  = $body.checks.smtp.status
               $expir = $body.checks.expiryService.status
               Write-Ok ("Health OK -- AD: {0}, SMTP: {1}, ExpiryService: {2}" -f $ad, $smtp, $expir)
           } catch {
               Write-Warning ("Health endpoint returned 200 but JSON parse failed: {0}" -f $_.Exception.Message)
               Write-Ok "Health OK (body could not be parsed — status 200 accepted)"
           }
       } else {
           Write-Step "Skipping post-deploy health check (-SkipHealthCheck specified)"
       }
       # ----- /STAB-019 -----
       ```
    3. Confirm the output text uses ONLY ASCII characters — no emoji (D-08). The checkmark indicator is `Write-Ok` (existing helper), the failure is `Write-Error`, and the body text uses the literal `--` separator (not an em-dash).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| installer script → newly deployed PassReset app | Local loopback or host-internal HTTP(S); installer trusts its own just-deployed output |
| installer exit code → CI / operator harness | Installer's exit code is the sole truth signal — a fail-open here silently advertises a broken deployment as success |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-02-01 | Elevation of Privilege / Tampering | Installer fail-open on health probe | mitigate | D-07: hard-fail (`exit 1`) when `/api/health` does not return 200 after 10 retries. No retry-forever loop, no warning-only path. `acceptance_criteria` greps the explicit `exit 1` presence. |
| T-10-02-02 | Denial of Service / Availability | `-SkipHealthCheck` misuse hides broken deploys | accept | D-10: switch defaults to `$false`; operators must explicitly opt-in. Documented in HUMAN-UAT Scenario C as "air-gapped only" path. |
| T-10-02-03 | Information Disclosure | Last response body printed on failure could leak internal hostnames | accept | The body is already the `/api/health` response, which per plan 10-01 contains no secrets (T-10-01-01 already mitigated). Hostname disclosure is local-only and already in the installer's own logs. |
| T-10-02-04 | Denial of Service | Blocking retry loop (20s worst case) under `-Force` CI install | accept | 20s worst-case wait is intentional and matches AppPool cold-start. Alternative (skip under -Force) would violate D-07. |
| T-10-02-05 | Tampering | Script parse regression (cf. STAB-005 MissingEndCurlyBrace) | mitigate | Acceptance criterion requires `pwsh -NoProfile -Command '[scriptblock]::Create(...)'` parse check to succeed. This is the same gate used for STAB-005. |
</threat_model>
